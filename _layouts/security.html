Reporting of CVEs and Security Issues

The WildFly community and our sponsor, Red Hat, take security bugs very seriously

We aim to take immediate action to address serious security-related problems that involve our projects. Note that we will only fix such issues in the most recent major release of WildFly.

Reporting of Security Issues

When reporting a security vulnerability it is important to not accidentally broadcast to the world that the issue exists, as this makes it easier for people to exploit it. The software industry uses the term embargo to describe the time a security issue is known internally until it public knowledge.

Our preferred way of reporting security issues in WildFly and its related projects are listed below.

Email the mailing list

The list at XXX@redhat.com is the preferred mechanism for outside users to report security issues. This A member of the WildFly team will open the required issues.

Open a Jira issue

If you are an experienced contributor, and choose to open a Jira issue in the correct project at https://issues.redhat.com, make sure to set the Security Level field to Embargoed Security Issue. This keeps it private.

Note that if you do open a Jira issue with the Security Level set to Embargoed Security Issue, you will not be able to view the issue due to our security policy.

Other considerations

It might be tempting to attempt to fix the security issue, especially if it seems like something easy. If you do do this, please make sure that you do not open any public pull requests since that risks breaking the embargo.

Instead, make a private fork of the repository, and send us a message via XXX@redhat.com. A member of the team will be in touch. Once that happens, you will need to add them as a contributor to your private fork so that they can see your changes.
